## Open Issues

### FINDING-1: Missing Smooth Scroll Interpolation Layer

**Severity:** Critical  
**Reference section:** Layer 1 — Lenis Smooth Scroll  
**File(s):** [src/app/page.tsx](file:///d:/NSBM/Projects%202.0/KaprukaNewDesign2.0/src/app/page.tsx#L156-L210), [package.json](file:///d:/NSBM/Projects%202.0/KaprukaNewDesign2.0/package.json#L15)  
**Status:** Missing  

**What the reference requires:**  
Lenis smooth scroll should intercept wheel/touch events and apply linear interpolation (lerp) to the scroll position. It must be bridged with GSAP's ticker (`gsap.ticker.add`) and configure `lagSmoothing(0)` to prevent the browser's raw, jagged scroll updates from causing micro-stutters in the animation timeline.

**What your codebase has:**  
Although `lenis` is installed as a package dependency, it is completely absent from the codebase. The scroll behavior is handled by a native, passive scroll listener on the `#main-scroll-container` `div` (`overflow-y: auto`), which directly sets Framer Motion values.

**Gap / Problem:**  
Without scroll interpolation, scrolling is subject to the browser's raw, un-smoothed delta updates. When mapped directly to scroll-driven properties, this creates visible steps and micro-stutters rather than a fluid 60fps experience.

**Compatibility flag:** None (Refactoring layout to window-level scrolling resolves native container restrictions).  

**Proposed resolution:**  
Import and initialize Lenis in `src/app/page.tsx` on window scroll. Bridge its update loop with GSAP's ticker, disable lag smoothing, and bind its scroll events to update ScrollTrigger.

### FINDING-2: Missing ScrollTrigger Orchestration Layer

**Severity:** Critical  
**Reference section:** Layer 3 — ScrollTrigger Scrubbing  
**File(s):** [src/app/page.tsx](file:///d:/NSBM/Projects%202.0/KaprukaNewDesign2.0/src/app/page.tsx#L156-L238)  
**Status:** Missing  

**What the reference requires:**  
GSAP ScrollTrigger must orchestrate scroll-linked timelines using `scrub: true` (or `scrub: 0`) and linear easing (`ease: 'none'`) to link the scroll position directly and linearly to the target animation properties.

**What your codebase has:**  
All scroll-linked transformations (`heroY`, `sec2LeftY`, `sec2RightY`, etc.) are computed via Framer Motion's `useTransform` reading from `scrollYProgress` values generated by the native scroll handler.

**Gap / Problem:**  
Framer Motion is not optimized for scrubbing high-frequency properties like a video's `currentTime`. React state updates and MotionValue changes are out of phase with browser painting and video decoding, leading to visible tearing and lag.

**Proposed resolution:**  
Install `gsap` and `@types/gsap`, register the `ScrollTrigger` plugin, and build a unified scrubbing timeline that drives the scroll animations.

### FINDING-3: Canvas-Based Frame Sequence Instead of Video Scrubbing

**Severity:** Critical  
**Reference section:** Layer 4 — Video currentTime Scrubbing  
**File(s):** [src/app/page.tsx](file:///d:/NSBM/Projects%202.0/KaprukaNewDesign2.0/src/app/page.tsx#L92-L154), [src/components/Preloader.tsx](file:///d:/NSBM/Projects%202.0/KaprukaNewDesign2.0/src/components/Preloader.tsx#L14-L96)  
**Status:** Incorrect  

**What the reference requires:**  
The scroll animation must be achieved by scrubbing the `currentTime` of a `<video>` element with keyframe-accurate seeking, utilizing hardware decoding. The setup must include attributes `muted`, `playsinline`, `preload="metadata"`, a mobile fallback (autoplay loop), and an iOS/Safari priming execution (`video.play().then(() => video.pause())`).

**What your codebase has:**  
The background animation uses a custom Canvas engine. During the preloader phase, 300 separate JPEG image frames are fetched, decoded off-thread into `ImageBitmap` objects, and stored on `window.preloadedFrames`. On scroll, the frame index is calculated synchronously, and `ctx.drawImage` renders the bitmap onto a `<canvas>` element.

**Gap / Problem:**  
While `ImageBitmap` drawing is fast, keeping 300 raw decoded bitmaps in memory is extremely RAM-intensive (consuming ~45MB–60MB of browser heap memory depending on resolution). It also requires a long loading time (blocking the user behind a preloader). A native video element uses the GPU's hardware video decoder, resulting in negligible memory overhead and zero preloader latency.

**Proposed resolution:**  
Refactor `src/app/page.tsx` to replace the `<canvas>` element with a `<video>` element referencing `/assets/video/heroScreen.mp4`. Bind `currentTime` to the ScrollTrigger timeline and implement the iOS priming sequence.

### FINDING-4: Pinning Structure Mismatch

**Severity:** Medium  
**Reference section:** Pinning — Do Less, Not More  
**File(s):** [src/app/page.tsx](file:///d:/NSBM/Projects%202.0/KaprukaNewDesign2.0/src/app/page.tsx#L556-L1029)  
**Status:** Partial  

**What the reference requires:**  
To pin the background scroll-scrubbed asset smoothly, the markup must use GPU-composited native `position: sticky` on the video container with a tall scrolling wrapper (e.g. `500vh`) to provide range. This avoids expensive coordinate recalculations associated with JS-based pinning.

**What your codebase has:**  
The background `<canvas>` is styled as `fixed inset-0 z-0` and does not participate in layout flow. The scrolling page sections scroll naturally over it. The scroll container `#main-scroll-container` has a fixed height of `100vh` and child sections are styled with `h-screen`.

**Gap / Problem:**  
If we switch to a video element, keeping the background `fixed inset-0` functions similarly to `position: sticky` in terms of screen lock. However, we must ensure that the scrolling overlay sections are correctly positioned and that their entry/exit points are synchronized with the timeline segments of the video.

**Proposed resolution:**  
Retain the GPU-safe `fixed inset-0` positioning for the background `<video>` container (avoiding the need for complex ScrollTrigger pin spacers) and set up the outer container scroll height to map to the video duration.

### FINDING-8: Library Conflict Between Framer Motion and GSAP

**Severity:** High  
**Reference section:** Library Conflicts  
**File(s):** [src/app/page.tsx](file:///d:/NSBM/Projects%202.0/KaprukaNewDesign2.0/src/app/page.tsx#L20-L21), [package.json](file:///d:/NSBM/Projects%202.0/KaprukaNewDesign2.0/package.json#L14)  
**Status:** Incorrect  

**What the reference requires:**  
Competing scroll listeners or animation libraries driving the same elements/events must be avoided to prevent competing updates and synchronization jitter.

**What your codebase has:**  
Currently, Framer Motion drives the floating animations for all sections on scroll, while we plan to introduce GSAP ScrollTrigger to scrub the video.

**Gap / Problem:**  
Running both Framer Motion scroll-driven transforms and GSAP ScrollTrigger updates concurrently will cause them to compete for frame updates, introducing micro-stutters. To achieve a perfectly synchronized visual layout (similar to GTA VI), the position changes of the floating panels and the video frame scrubbing must be coordinated by the same engine.

**Proposed resolution:**  
Migrate all scroll-driven container animations (the rising, fading, and panning of the panels) from Framer Motion `useTransform` to the GSAP ScrollTrigger timeline. Framer Motion should only be retained for standard component-level animations (like state switches, Cart Drawer sliding, and Chat bubble entry) that are not linked to the scroll position.

### FINDING-6: Video Asset Encoding GOP Size Issues

**Severity:** High  
**Reference section:** Video Encoding — The Most Overlooked Part  
**File(s):** `public/assets/video/heroScreen.mp4`  
**Status:** Resolved  

**What the reference requires:**  
A scroll-scrubbed video must be encoded with a GOP (Group of Pictures) size of 1 (`-g 1`), making every frame a keyframe (All-Intra). It should also have `-movflags faststart` applied (moving the `moov` atom to the beginning of the file) and match the container width.

**What your codebase has:**  
The original asset `heroScreen.mp4` used inter-frame compression (`B` and `P` frames present throughout, meaning GOP > 1) which causes seeking stutters due to decoding latency.

**Resolution:**  
The asset has been successfully re-encoded using FFmpeg with the parameters:
`ffmpeg -y -i public/assets/video/heroScreen.mp4 -vf scale=1280:-1 -an -movflags faststart -vcodec libx264 -crf 20 -g 1 -pix_fmt yuv420p public/assets/video/heroScreen_g1.mp4`
This strips the unused audio channel, forces every frame to be a keyframe, scales the resolution to match layout widths, and writes the `moov` atom at the file start. The file has replaced the original asset at `/public/assets/video/heroScreen.mp4`.

## Known Incompatibilities

*   **GSAP spacer pinning vs. Responsive Grid Layouts:**  
    The Rockstar website uses GSAP's `pin: true` which inserts layout spacer wrapper divs. In our React split-screen dashboard workspace, layout spacers can disrupt flexbox alignments and container spacing.  
    *Resolution:* We avoid ScrollTrigger container pinning and instead place the `<video>` in a native `fixed inset-0 z-0` container. We then stretch the document height to `400vh` to supply scroll range, allowing sections to scroll naturally on top of the fixed background.
